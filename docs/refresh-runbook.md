# コンテンツ更新runbook

`ai-primer` の鮮度維持は「自動検出 + 人間が更新」の二段構えです。**本文の自動更新は行いません**（機械的に再生成できる性質のデータがないため）。このドキュメントは、月次の `freshness-report.yml` が Issue を作った後に、実際の内容を更新するための手順です。

## 自動化されている範囲（`freshness-report.yml` / `scripts/check-freshness.ts`）

1. 全レッスン・モデルカタログ・年表・用語集の `sources[].url` を重複排除し、疑わしいリンク（HTTPエラー・タイムアウト）を検出。
2. `lastVerified` が90日を超えた項目を列挙。
3. 上記いずれかがあれば、GitHub Issue「鮮度レポート: 要確認項目あり」を作成/更新。

**手動実行**: `npm run check-freshness`（ローカルでも確認可能）。

## 人間 + Claude Code セッションでの更新手順

Issueを見つけたら、以下の手順で対応します。

### 1. 対象を確認する
Issue本文のリストから、以下を確認する:
- リンク切れの疑いがある項目 → **実際にブラウザで開いて確認する（必須）**。`check-freshness.ts` は単純な `fetch()` を使うため、Britannica・OpenAI・Perplexity・Midjourneyなど一部のサイトは実ブラウザの UA でも `fetch()` 単体はボット判定で 403 を返すことが確認済み（2026-07-15 検証）。**スクリプトが「疑わしい」と報告した実績のうち複数件が実ブラウザでは正常 200 だった**ため、リストは足切りの参考程度に扱い、必ず目視確認してから対応する。
- 90日超で未確認の項目 → 内容が今も正しいか確認する必要がある。

### 2. Claude Codeセッションで検証する
対象のレッスン/モデル/年表項目について:
- **Web検索で現在の事実を確認する**（本サイトの執筆時と同じ要領。AGENTS.mdの「コンテンツ執筆規約」参照）。
- リンク切れの場合、後継URLや同等の公式情報源を探して差し替える。
- 内容が古くなっている場合（例: サービスが終了した、名称が変わった）、本文を更新する。
- 変わっていなければ、`lastVerified` の日付だけを今日の日付に更新する。

### 3. 揮発性の高い情報は個別ページに寄せる
モデル名・料金など変わりやすい事実は、レッスン本文ではなく `src/engine/content/models.ts` に集約する方針を維持する（AGENTS.md参照）。レッスン本文を編集する前に、その情報が models.ts に属するべきでないか確認する。

### 4. テストとビルドを通す
```bash
npm run typecheck
npm run coverage   # engine/i18n層100%ゲート
npm run build
```
`content.test.ts` が id一意性・出典形式・`lastVerified` の妥当性を自動検証する。

### 5. コミット・PR
1コミット=1項目更新を基本とする（AGENTS.mdのコミット粒度規約）。まとめて直す場合も、何を確認・変更したかコミットメッセージに明記する。

## なぜ自動draft PRを作らないのか

chronoscroll（歴史年表アプリ）は「新規データを機械的に取得→ビルド→PR」という完全自動フローを持つが、これは外部APIから構造化データを再取得できるため成立する。`ai-primer` は事実確認そのものに人間/AIの判断（Web検索・一次情報源の読み込み）が必要なため、空のdraft PRを量産するより、**Issueで気づきを提供し、人間が着手する**方が誠実で実効性がある。
